Final confirmed rules
1. Attendee access flow
Attendees must create an account.

After login, they are redirected directly to the specific event details page.

No global event list is needed for MVP.

2. Purchase-before-seat-selection flow
The attendee does not choose exact seats before payment.

Flow:

Buyer selects ticket type and quantity.

System checks available capacity.

System reserves ticket quantity/capacity only.

Buyer pays through HitPay.

HitPay webhook confirms payment.

Buyer can assign exact seats immediately or later.

QR tickets are only generated after seat assignment is complete.

3. Capacity reservation rule
Before payment, the system reserves ticket quantity, not specific seats.

If there are not enough available ticket slots, the user cannot proceed to payment.

4. Ticket type and seat inventory mapping
Ticket type quantity must directly match seat inventory.

Example:

Organizer creates 10 VIP tickets.

System creates or maintains exactly 10 VIP seat records.

VIP buyers can only choose from VIP seats.

This rule prevents overselling.

5. Seat creation model
Use a hybrid seat setup:

System auto-generates seats from ticket quantity.

Organizer can rename or edit seat labels manually.

Seating can be table-based or generic.

Example table-based:

Table 1 - Seat 1

Table 1 - Seat 2

Example generic:

VIP-001

VIP-002

6. Early Bird pricing
Early Bird applies to:

discounted price

reserved ticket quantity

It does not reserve exact seats.

If the payment window expires:

Early Bird price is no longer valid.

Price reverts to regular price.

No selected seat is lost because exact seats were not selected yet.

7. Dynamic timeout settings
Organizers can configure event-level timeout settings, including:

payment hold duration

capacity reservation duration

Early Bird price hold duration

These should not be hardcoded.

8. Group registration
A buyer can purchase multiple tickets.

Example: buyer purchases 5 tickets.

After payment, buyer must assign:

5 unique seats

5 attendee names

5 attendee email addresses

The buyer may assign one ticket to themselves and the rest to guests.

Guests do not need Eventuz accounts.

9. QR ticket generation and email delivery
QR tickets are generated only after:

payment is confirmed

exact seats are selected

attendee names are assigned

attendee emails are provided

Each assigned attendee receives their own QR e-ticket by email.

Tickets are also downloadable after successful assignment.

Super Admin configures SMTP settings.

Key insight
The MVP should be modeled around paid seat rights, not immediate seat ownership.

That means the system has two separate stages:

Stage 1: Purchase right
The buyer pays for a number of ticket rights.

Status: paid_unassigned

Stage 2: Seat assignment
The buyer converts those paid rights into actual named QR tickets.

Status: assigned_ticket_issued

This separation is the heart of Eventuz’s logic.

Ticket quantity and seat inventory must stay synchronized.

If Organizer edits ticket quantity, the system must handle seat generation carefully.

Tickets must never be issued from frontend success redirects alone.

6. Business rules
Payment and ticket rules
Tickets must never be issued before successful HitPay webhook confirmation.

Frontend success redirect is not enough to mark payment as paid.

Payment confirmation must come from verified webhook.

One paid ticket equals exactly one seat.

QR ticket is generated only after:

payment is confirmed

exact seat is selected

attendee name is provided

attendee email is provided

Each QR ticket must have a secure unique token.

A QR ticket can only be checked in once.

Capacity and seat rules
Ticket type quantity must match seat inventory count.

A buyer can only pay if enough capacity exists.

Before payment, the system reserves capacity only.

Before payment, exact seats are not selected.

After payment, buyer selects exact seats.

Buyer can assign seats immediately or later.

Paid but unassigned orders must reduce available capacity.

Assigned seats are no longer available to others.

Decreasing ticket quantity should be blocked if affected seats are already assigned or paid against.

Increasing ticket quantity should generate additional seat records.

Early Bird rules
Early Bird discount applies to price only.

Early Bird does not reserve exact seats.

Early Bird price is held only within organizer-configured time limit.

If buyer fails to pay within the configured time limit, price reverts to regular price.

Early Bird eligibility should be calculated at checkout/order creation.

Final paid amount must match the valid price at payment confirmation.

Seat assignment rules
Buyer must assign exactly the number of seats purchased.

Buyer may assign one seat to self and others to guests.

Each assigned seat requires:

attendee name

attendee email

Assigned guests do not need accounts.

QR tickets are emailed to assigned attendee emails, not only the buyer.

Partial assignment may be allowed, but ticket issuance only happens per fully assigned ticket.

Order should show partially_assigned if not all paid seats are assigned.

Recommended MVP simplification:

Allow saving partial assignment, but only generate QR tickets for completed assigned seats.

Alternative stricter version:

Require all seats to be assigned before generating any QR tickets.

I recommend the stricter version for MVP to reduce edge cases.

Staff rules
Organizer can invite staff by email.

Staff can scan tickets for assigned event only.

Staff cannot edit event settings.

Staff cannot view full revenue dashboard.

Staff scan activity should be logged.

Super Admin rules
Super Admin can configure SMTP.

Super Admin can monitor events and users.

Super Admin can disable events.

Super Admin cannot edit event details.

Disabled events should block new registrations.

Existing ticket validity after event disablement should be a deliberate admin decision later; for MVP, disabling should block public access and new purchases.

Recommended operating process
How to use these prompts
Use this workflow for every step:

Paste the global instruction.

Paste the current step prompt.

Let Bolt implement.

Test manually.

Ask Bolt to fix only the failing part.

Move to the next step only when acceptance criteria pass.

Do not stack multiple prompts together.

Best recommendation
Build only up to Step 5 first before touching attendee or payment flows.

Your first milestone should be:

Organizer can create an event, create ticket types, and generate/edit matching seat inventory.

That is the foundation of the whole system.

If ticket quantity and seat inventory are wrong, every later feature becomes fragile.

Tips for working with Bolt
1. Always force inspection first
Use this before any implementation:

Before making changes, inspect the current project structure, routes, database/schema, components, and existing logic. Then summarize what you found and only after that implement the requested feature.
This reduces random changes.

2. Never ask for “full system”
Avoid:

Build the full Eventuz app.
Use:

Implement only Phase 1: organizer event setup, ticket types, and seat inventory generation.
3. Keep each feature testable
Bad prompt:

Add payments, QR tickets, emails, and dashboard.
Better:

Only implement HitPay webhook confirmation. Do not build QR tickets or email delivery yet.
4. Protect the database logic
Your highest-risk rules are:

ticket type quantity must match seat count

paid capacity must reduce availability

seats are selected only after payment

tickets are generated only after seat assignment

payment success comes only from webhook

Repeat these often in prompts.

5. Use rollback instead of repair marathons
If Bolt makes a bad broad change, do not keep patching endlessly.

Use:

Rollback the last change. Restore the previous working state. Then implement only the requested feature with minimal changes.
6. Ask for changed files every time
After implementation, ask:

List every file changed, explain why it changed, and provide a test checklist for this feature.
This helps you catch scope creep.

7. Test status transitions aggressively
Eventuz depends on statuses.

Critical order transitions:

capacity_held → payment_pending → paid_unassigned → completed
Critical ticket transition:

issued → checked_in
Critical seat transition:

available → assigned → checked_in
If statuses are messy, the whole app becomes hard to debug.

Risks / trade-offs
Main risk
The system has complex state transitions. If you let Bolt build too many phases at once, it may mix payment, seat assignment, and ticket issuance incorrectly.

Trade-off
The pay-first-seat-later model is safer against unpaid seat blocking, but it requires clear UI messaging and strong dashboard visibility for “Paid but Unassigned.”

Biggest technical warning
Do not let frontend logic decide payment success.

Only the backend/webhook should move an order to:

paid_unassigned

Start with Step 1, then Step 2, then Step 3–5.

Your first real milestone is:

Organizer foundation complete: event setup, ticket types, and synchronized seat inventory.

Once that milestone works, move to attendee orders and payments.
